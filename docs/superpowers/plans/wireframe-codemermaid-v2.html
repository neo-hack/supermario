concept

Token 提取与验证

这个 middleware 从 Authorization header 里提取 Bearer token。注意它用 optional chaining — 没有 header 时不 crash，只是拿到 undefined。

提取后调用 verify() 验证签名和过期时间。验证通过就把 user 对象挂到 context 上，后续 handler 直接用。

concept

这个 middleware 不 throw

大多数 auth middleware 会在 token 无效时抛 401。但这里不 — 它设 c.set('user', null) 然后 next()。下游 handler 自己决定 null user 够不够用。这意味着同一个 middleware 同时服务公开路由和受保护路由。

quiz
当 token 无效（过期/签名错误）时，这个 middleware 做了什么？
A 抛出 401 Unauthorized 错误
B 设置 user 为 null 并继续执行
C 重定向到登录页
D 返回空响应
正确：B — middleware 在 catch 块里设 c.set('user', null) 然后 next()。不 throw、不 redirect、不中断请求。这是一个有意的设计决策 — 让下游 handler 决定如何处理未认证用户。

逐行解析 auth middleware

src/middleware/auth.ts TypeScript 
1export const auth: Middleware = async (c, next) => {2  const token = c.req.header('Authorization')?.slice(7);3  if (!token) { c.set('user', null); return next(); }4  try {5    const user = await verify(token);6    c.set('user', user);7  } catch {8    c.set('user', null);9  }10  return next();11};
L1-2

从 Authorization header 提取 Bearer token。Optional chaining 保证没 header 时也不 crash。

L5

verify() 验证 JWT 签名和过期时间。抛异常说明 token 无效。

L8

catch 块不 throw — 设 null user，让请求继续走。

L10

无论成功失败都 next()。这就是"不阻断"的精髓。

takeaway

这个 auth middleware 的核心设计是 标注而非阻断 — 它验证 token、把 user 挂到 context 上，但从不拒绝请求。每个下游 handler 自己决定 null user 是公开访问还是 401。这种模式让一个 middleware 同时服务公开和受保护路由。
