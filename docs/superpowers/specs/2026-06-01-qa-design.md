# QA Skill Design

## Overview

A supermario agent skill that systematically tests web pages using agent-browser. Two modes: free exploration (no qa.md) discovers bugs via 7-item checklist, case verification (with qa.md) validates expected behaviors defined by the user. Single-page by default, multi-page via qa.md scenarios. No issue limit — tests until all interactive elements are covered. Outputs Markdown report + Apple-style HTML report + 8-dimension health score.

Does NOT fix bugs. Discovery and reporting only.

## Inputs

| Parameter | Required | Default | Description |
|-----------|----------|---------|-------------|
| URL | Yes (free mode) | - | Target page URL |
| qa.md | No | - | Test case file with scenarios |
| Output directory | No | `./qa-output/` | Where to save artifacts |

## Two Modes

### Free Exploration (no qa.md)

- Single page, exhaust all interactive elements
- No issue count limit — test until every element is visited
- Judge each interaction against 7-item checklist + issue taxonomy
- Scroll reveals new elements → add to queue, continue
- 8-dimension health score at the end

### Case Verification (with qa.md)

- Parse `<scenario>` blocks from qa.md
- Execute each `<action>` and verify `<expect>`
- PASS / FAIL binary judgment — failures get screenshot evidence
- Scenarios can span pages via `url` attribute on `<scenario>`
- After all scenarios complete, free-explore any elements not covered by scenarios
- Same 8-dimension health score

## qa.md Format

Human-writable, loose natural language format:

```markdown
<scenario name="登录流程" url="/login">
  <action>填写 Email 为 test@example.com</action>
  <expect>无报错</expect>

  <action>填写 Password 为 wrong-password</action>
  <expect>无报错</expect>

  <action>点击 "Sign In" 按钮</action>
  <expect>出现错误提示 "邮箱或密码不正确"</expect>
</scenario>

<scenario name="创建任务" url="/tasks/new">
  <action>填写 Title 为 "测试任务"</action>
  <expect>无报错</expect>

  <action>点击 "Submit" 按钮</action>
  <expect>跳转到任务列表页，新任务出现在列表顶部</expect>
</scenario>
```

Processing:
1. Parse all `<scenario>` blocks
2. For each scenario: `agent-browser goto <url>`, then execute action/expect pairs sequentially
3. Match action text to agent-browser `@eN` elements via semantic matching (role + label)
4. After action, LLM judges whether `<expect>` holds based on snapshot + screenshot
5. PASS → continue. FAIL → record ISSUE-NNN with screenshot evidence

### qa.md Creation After Run

After the QA run completes (both modes), ask the user:

> "是否将本次探索结果更新到 qa.md？"

- **Create new qa.md**: scaffold all discovered interactive elements as `<action>` tags with actual observed behavior written into `<expect>`
- **Update existing qa.md**: append newly discovered elements that weren't in the original qa.md
- User can decline — qa.md is purely optional

## Core Flow

```
Initialize → Orient → Execute Scenarios (if qa.md) → Free Explore → Score → Report → Cleanup → Ask about qa.md
```

### Initialize

```bash
mkdir -p {OUTPUT_DIR}/screenshots
agent-browser open {URL}
agent-browser wait --load networkidle
```

### Orient

```bash
agent-browser screenshot --annotate {OUTPUT_DIR}/screenshots/initial.png
agent-browser snapshot -i
agent-browser console
agent-browser errors
```

Count and classify interactive elements by role. Note distribution in report header.

### Execute Scenarios (if qa.md present)

For each `<scenario>`:

1. `agent-browser goto <url>` if scenario has a URL
2. `agent-browser wait 1000`
3. For each action/expect pair:
   - Screenshot before
   - Execute action (semantic match to `@eN` element)
   - `agent-browser wait 1000`
   - Screenshot after
   - `agent-browser snapshot` to check what changed
   - `agent-browser console` to check for errors
   - Judge expect: PASS or FAIL
   - If FAIL: assign ISSUE-NNN, record evidence
4. Log result in report

### Free Explore

If no qa.md: explore all interactive elements on the page.
If qa.md: after scenarios complete, explore elements not covered by any scenario.

Per-element workflow:

1. Screenshot before
2. Execute operation by ARIA role (click/fill/select/toggle)
3. `agent-browser wait 1000`
4. Screenshot after
5. `agent-browser snapshot` to check what changed
6. `agent-browser console` + `agent-browser errors`
7. Judge against 7-item checklist + issue taxonomy
8. If issue found: assign ISSUE-NNN, write to report immediately
9. Continue to next element

Stopping condition: all interactive elements on the page have been explored. Scroll reveals new elements → discover with `agent-browser snapshot -i`, add to queue, continue.

### 7-Item Per-Page Checklist

1. **Visual scan** — annotated screenshot for layout issues
2. **Interactive elements** — click every button/control, check response
3. **Forms** — fill and submit, test empty/invalid/boundary inputs
4. **Navigation** — check paths in and out (note but don't follow external links)
5. **States** — empty state, loading, error, overflow
6. **Console** — JS errors and failed requests after each interaction
7. **Accessibility basics** — missing labels, keyboard-unreachable elements

### 8-Dimension Health Score

Weighted average, each category starts at 100, deduct per finding:

| Category | Weight | Scoring |
|----------|--------|---------|
| Console | 15% | 0 errors=100, 1-3=70, 4-10=40, 10+=10 |
| Links | 10% | each broken link -15 (min 0) |
| Visual | 10% | critical -25, high -15, medium -8, low -3 |
| Functional | 20% | same scale |
| UX | 15% | same scale |
| Performance | 10% | same scale |
| Content | 5% | same scale |
| Accessibility | 15% | same scale |

Final score = Σ(category_score × weight)

Save as `baseline.json` for cross-run comparison.

### Issue Severity

| Level | Definition |
|-------|------------|
| Critical | Blocks core workflow, causes data loss, or crashes |
| High | Major feature broken, no workaround |
| Medium | Feature works with noticeable problems, workaround exists |
| Low | Minor cosmetic or polish issue |

### Evidence & Screenshots

Both modes use the same screenshot protocol — every action gets a before/after pair:

```
screenshots/
  step-001.png          # before action
  step-001-after.png    # after action
  step-002.png
  step-002-after.png
  ...
```

- Every action gets before + after screenshots, regardless of PASS/FAIL
- PASS results still have full screenshot evidence (enables cross-run visual diff)
- FAIL results additionally get an annotated screenshot highlighting the failure

### Evidence Tiers

- **Interactive bugs**: before screenshot → action → after screenshot → snapshot diff
- **Static bugs**: single `screenshot --annotate`

### Report Structure

Reports differ between modes.

**Free exploration mode:**

```
# QA Report: {URL}

## Session Info
## Health Score (8-dimension radar)
## Exploration Log
  ### Step 1: Click @e3 "Submit" button
  ### Step 2: Fill @e5 "Email" input
  ...
## Issues
  ### ISSUE-001: ...
  ### ISSUE-002: ...
## Summary
```

**Case verification mode:**

```
# QA Report: {URL}

## Session Info
## Health Score (8-dimension radar)
## Scenario Results
  | Scenario | Total | PASS | FAIL |
  |----------|-------|------|------|
  | 登录流程  | 3     | 2    | 1    |
  | 创建任务  | 2     | 2    | 0    |

## Scenario Details
  ### Scenario: 登录流程
    <action>填写 Email</action>  ✅ PASS
    <action>填写 Password</action>  ✅ PASS
    <action>点击 Sign In</action>  ❌ FAIL — 期望 "邮箱或密码不正确"，实际无任何提示

## Uncovered Elements (free exploration)
  ### Step N: Click @e7 "Remember me" checkbox
  ...

## Issues
  ### ISSUE-001: Sign In 无错误提示
  ### ISSUE-002: ...

## Summary
```

### Report Output

```
{OUTPUT_DIR}/
  report.md                      # Markdown report
  report.html                    # Apple-style HTML report (DESIGN.md)
  baseline.json                  # Health score baseline
  screenshots/
    initial.png
    issue-001-step-1.png
    issue-001-result.png
    ...
```

### Cleanup

After exploring all elements:

1. Compute final health score
2. Re-read report, update summary counts to match actual issues
3. Generate HTML report from Markdown + screenshots
4. `agent-browser close`
5. Ask user about qa.md update

## agent-browser Commands Used

| Phase | Command | Purpose |
|-------|---------|---------|
| Setup | `agent-browser open <url>` | Launch browser |
| Setup | `agent-browser wait --load networkidle` | Wait for page ready |
| Orient | `agent-browser screenshot --annotate <path>` | Annotated initial screenshot |
| Orient | `agent-browser snapshot -i` | Discover interactive elements |
| Explore | `agent-browser snapshot` | Full tree for context |
| Explore | `agent-browser click @eN` | Click element |
| Explore | `agent-browser fill @eN "text"` | Fill input |
| Explore | `agent-browser select @eN "value"` | Select dropdown option |
| Explore | `agent-browser wait 1000` | Let page settle |
| Explore | `agent-browser screenshot <path>` | Capture state |
| Explore | `agent-browser console` | Check JS errors |
| Explore | `agent-browser errors` | Check exceptions |
| Explore | `agent-browser goto <url>` | Navigate (scenario mode) |
| Cleanup | `agent-browser close` | Close browser |

## Constraints

- Single page in free mode, multi-page via qa.md scenarios only
- No source code reading — judge only from browser observations
- Fully automated — no user confirmation during exploration
- `agent-browser` only — no Playwright, Puppeteer, or other tools
- Does NOT fix bugs — discovery and reporting only
- `--profile Default` must come BEFORE the `open` subcommand

## Relationship to Existing Skills

- **dogfood** (agent-browser): upstream inspiration. This skill forks the exploration logic, adds health scoring, HTML reports, qa.md case verification, and removes the 5-10 issue limit.
- **ux-explore**: complementary. QA finds bugs (broken things), ux-explore finds interaction intuition issues (works but confusing). Same Apple-style HTML report format.
- **ui-review**: compares live UI against design references. QA is autonomous exploration without reference designs.
