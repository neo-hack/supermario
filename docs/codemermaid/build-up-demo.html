concept

Start with the visible transcript

For chat-kit, the smallest useful surface is not the composer or the transport layer. It is a message appearing in the transcript. The Build-Up route should therefore start with how an SDKMessage becomes a visible card, then add stream merging, provider context, resources, permissions, and finally the composer.

line 1

Meta messages get their own branch before normal user/assistant content. That tells the reader this router is a typed boundary, not a generic list renderer.

line 5

The user branch filters synthetic/tool-result-only messages before rendering. Build-Up should explain this before adding turn collapse or time-travel actions.

line 17

UserMessage receives host slots and status props here. This is where chat-kit lets the host add retry/edit behavior without changing the message card itself.

line 28

The assistant branch is deliberately separate. Once this is clear, the course can introduce tool-use grouping inside AssistantMessage as the next capability.

Check the Build-Up rule

In this chat-kit Build-Up path, why is MessageRouter a better first code-walk than ChatComposer?
B is correct. MessageRouter receives one SDKMessage and chooses the visible leaf component. ChatComposer matters later, but it is not the smallest route from streamed data to transcript UI.
takeaway

For chat-kit, a strong Build-Up page would start at MessageRouter, then layer in threadStreamReducer, ToolResultProvider, ChatPanel layout, resource panels, permissions, and ChatComposer. That order teaches the visible transcript first, then explains the infrastructure that keeps it live and interactive.
